## Lifecycle Matrix

| Scenario | Behavior | Runtime shutdown boundary |
| --- | --- | --- |
| Window close | Hide main window to tray. | No shutdown request. |
| Titlebar close | Calls `hide_main_window`. | No shutdown request. |
| Tray show/hide | Shows or hides the main window only. | No shutdown request. |
| Tray quit / Settings Quit App | Sets explicit app-quit state and exits desktop process. | Proxy keeps running. |
| Attach existing runtime | UI remains attached/non-owning. | Normal app exit does not remote-stop it. |
| Stop Proxy | Requires exact owned/attached confirmation phrase. | Only explicit Stop Proxy may call runtime shutdown. |

## Native Close Smoke Follow-up

- A real Windows `WM_CLOSE` smoke initially found that the main window could still terminate the desktop process before the close listener was fully registered.
- Fixed by re-registering the native close handler after the frontend emits `codex-helper://window-ready`.
- Re-ran Windows smoke against `target/debug/codex-helper-desktop.exe` with an existing admin runtime on 127.0.0.1:4211:
  - `before_admin_4211=True`
  - `desktop_alive_after_wm_close=True`
  - `window_visible_after_wm_close=False`
  - `after_wm_close_admin_4211=True`
  - `after_desktop_process_cleanup_admin_4211=True`
- This proves a native close request hides the window and leaves both the desktop process and existing proxy runtime alive until the desktop process is explicitly cleaned up by the smoke script.
